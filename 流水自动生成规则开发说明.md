# 真诚流水自动生成规则开发说明

本文档用于指导 `SpeedEmulator` 后续实现流水自动生成功能。内容来自对真诚前台程序的只读结构分析与业务行为归纳，不复制反编译混淆代码，不涉及破解或绕过原软件数据库。

## 目标

在前台客户端中复刻“流水自动生成页面”的业务行为：

- 根据银行、用户、起止日期、金额配置生成流水明细。
- 参照明细和固定日期增加项按银行维护，所有用户共用同一银行规则数据。
- 生成结果写入本地库，后续可在流水明细页面查看、编辑、导入、导出。
- 生成后的余额必须满足用户设置的最后卡余额。

## 主要数据模型

### 用户配置

用户信息里需要保存自动生成相关配置：

- `StartTime`：开始日期
- `EndTime`：结束日期
- `InitialBalance`：期初余额
- `IsAutoInterest`：是否自动计算利息
- `GSelectIndex`：金额分配模式
- `GAllInMoney`：总进账金额
- `GAllOutMoney`：总出账金额
- `GLastMoney`：最后卡上余额
- `GMinInMoneyMonth1`、`GMaxInMoneyMonth1`：按总金额模式计算出的月收入范围
- `GMinOutMoneyMonth1`、`GMaxOutMoneyMonth1`：按总金额模式计算出的月支出范围
- `GMinInMoneyMonth2`、`GMaxInMoneyMonth2`：按月金额模式的月收入范围
- `GMinOutMoneyMonth2`、`GMaxOutMoneyMonth2`：按月金额模式的月支出范围

### 参照明细

参照明细是普通随机流水的模板来源，按银行保存。

核心字段：

- `IsCheck`：是否参与生成
- `BankId`：银行 ID
- `IncomeAttribute`：收入或支出
- `MinMoney`、`MaxMoney`：金额范围
- `FloutLength`：小数位
- `StartDay`、`EndDay`：生成时间小时范围
- `PercentMonth`：每月出现次数
- `TradeHoliday`：是否允许节假日交易
- `TradeWeekend`：是否允许周末交易
- 其它银行流水字段：日志号、地点、摘要、交易方式、交易渠道、交易说明、附言、对方账号、户名、对方开户行等

### 固定日期增加项

固定日期增加项按银行保存，用于在指定日期或日期规则上追加固定流水。

核心字段：

- `IsCheck`
- `BankId`
- `IncomeAttribute`
- `MinMoney`、`MaxMoney`
- `FloutLength`
- `StartDay`、`EndDay`
- `FixDay`：固定日期规则
- `ReCnt`：重复次数或数量区间
- `TradeHoliday`
- `TradeWeekend`
- 其它银行流水字段

### 月份明细配置

按月明细分配模式使用：

- `StartTime`
- `EndTime`
- `InMoney`
- `OutMoney`

它不是银行公共数据，而是生成配置的一部分。当前实现时可以先跟随页面配置保存。

### 生成流水

最终写入本地库的是流水记录：

- `BankId`
- `BankUserId`
- `AccountTime`
- `TradeMoney`
- `Balance`
- `IncomeFlag`
- `CreditType`
- 银行专属流水字段

生成过程中建议先使用中间模型，再映射成最终流水。

中间模型字段：

- `AccountTime`：发生时间
- `Amount`：金额，收入为正，支出为负
- `SourceType`：来源类型
- `SourceIndex`：来源行号
- `MoveFlag`：是否可移动

来源类型建议：

- `0`：参照明细
- `1`：固定日期增加项
- `-1`：利息流水

## 页面入口流程

用户列表页面点击“自动生成流水”后进入生成页面。

进入页面时：

1. 根据当前银行读取参照明细。
2. 根据当前银行读取固定日期增加项。
3. 根据当前银行读取利息配置。
4. 从当前用户读取起止日期、期初余额、总进账金额、最后卡余额、分配方式等配置。
5. 如果用户没有保存过配置，使用默认值。

点击“保存配置”：

1. 保存当前页面所有配置。
2. 同步 `StartTime`、`EndTime`、`InitialBalance` 到当前用户信息。
3. 保存成功必须弹窗提示。
4. 保存失败必须弹窗提示失败原因。

点击“开始生成”：

1. 校验至少有一条参照明细被勾选。
2. 保存当前配置。
3. 如该用户已有流水，提示是否覆盖旧数据。
4. 生成临时流水项。
5. 追加固定日期流水。
6. 追加利息流水。
7. 执行金额与余额校验。
8. 按时间排序。
9. 计算每条流水余额。
10. 映射成最终流水记录。
11. 保存本地库。
12. 打开流水明细页面。

## 金额分配模式

### 按总进账金额方式分配金额

输入：

- `StartTime`
- `EndTime`
- `AllInMoney`
- `LastMoney`
- `InitialBalance`

计算按钮行为：

```text
月份数 = 起始日期到结束日期覆盖的月份数，包含首尾月
月均收入 = 总进账金额 / 月份数
月均支出 = (总进账金额 - 最后卡余额) / 月份数

月收入最小金额 = round(月均收入)
月收入最大金额 = round(月均收入)
月支出最小金额 = round(月均支出)
月支出最大金额 = round(月均支出)
```

注意：真诚可见逻辑中，计算按钮没有使用期初余额。

### 按月进账金额方式分配金额

输入：

- 月收入最小金额
- 月收入最大金额
- 月支出最小金额
- 月支出最大金额

生成时按每个月分别控制收入、支出范围。

### 按月明细分配

输入：

- 每个配置段的开始月份
- 每个配置段的结束月份
- 入账金额
- 出账金额

生成时按配置段分别控制收入、支出目标。

## 参照明细生成规则

只处理 `IsCheck=true` 的参照明细。

每条参照明细生成规则：

1. 按月份循环。
2. 每月生成次数来自 `PercentMonth`。
3. 在当月随机选择日期和时间。
4. 小时范围使用 `StartDay` 到 `EndDay`。
5. 若不允许周末，遇到周末跳过。
6. 若不允许节假日，遇到节假日跳过。
7. 金额在 `MinMoney` 到 `MaxMoney` 间随机。
8. 小数位按 `FloutLength` 处理。
9. `IncomeAttribute=收入` 时金额为正。
10. `IncomeAttribute=支出` 时金额为负。
11. 收入累计不能超过总进账金额。

### 金额小数位规则

参照明细和固定日期增加项的 `FloutLength` 决定单笔流水金额精度。生成随机金额、按总额重新分摊金额、修正收入或支出总额时，都必须尽量保持来源规则的小数位粒度。

规则如下：

- `FloutLength` 为空时按 `0` 处理。
- `0`：取整数，例如 `1234.56` 生成或修正后为 `1235`。
- `-1`：取整到十位，例如 `1234.56` 生成或修正后为 `1230`。
- `-2`：取整到百位，例如 `1234.56` 生成或修正后为 `1200`。
- 正数：保留对应小数位，例如 `2` 保留两位小数，`3` 保留三位小数。

注意：如果目标总额本身带分，而所有可调整流水都要求整数、十位或百位，数学上无法同时满足“每条流水完全符合小数位规则”和“最终余额精确到分”。这种情况下应优先使用最小粒度的可调整流水配平；仍无法配平时，才允许最后一条可调整流水保留极小尾差，保证最后卡余额不多不少。

## 固定日期增加项生成规则

只处理 `IsCheck=true` 的固定日期增加项。

`FixDay` 规则建议按以下方式实现：

- 单个数字，例如 `21`：每月固定 21 日生成。
- 日期范围，例如 `5-10`：在每月 5 日到 10 日之间随机生成。
- 每日规则：在日期范围内每天生成。
- 随机日期规则：在当前月份有效区间内随机生成。
- 全局随机规则：按 `ReCnt` 在完整起止日期范围内生成指定数量。

固定项日期处理：

1. 先计算目标日期。
2. 如果日期超出生成起止范围，跳过。
3. 若不允许周末且目标日是周末，日期向后移动，直到满足条件。
4. 若不允许节假日且目标日是节假日，日期向后移动，直到满足条件。
5. 金额规则同参照明细。

## 利息生成规则

利息配置按银行保存。

固定字段：

- `Day`：结息日
- `Month`：结息月份，例如 `3;6;9;12`
- `StartHour`
- `EndHour`
- `Rate`：年利率百分比
- 其它流水字段默认值

生成步骤：

1. 根据结息月份和结息日生成利息占位流水。
2. 占位流水金额初始为 `0`。
3. 根据每日余额积数计算利息。
4. 利息金额写回占位流水。
5. 利息流水参与后续余额滚动。

利息计算：

```text
日积数 = sum(当日余额 * 持续天数)
利息 = round(日积数 * 年利率 / 36500, 2)
```

## 最终金额与余额校验规则

生成完成后必须执行以下校验。金额计算建议统一使用 `decimal`，最终保留两位小数。

### 收入合计限制

收入合计不能大于总进账金额。

```text
收入合计 = 所有 TradeMoney > 0 的金额之和
收入合计 <= 总进账金额
```

如果生成过程中即将超过总进账金额，应停止继续生成收入项，或重新选择金额，使收入合计不超过总进账金额。

### 支出与期初余额修正

最终余额公式必须成立：

```text
最后卡余额 = 期初余额 + 收入合计 - 支出合计
```

因此：

```text
总支出金额 = 期初余额 + 总进账金额 - 最后卡余额
```

如果生成后的支出合计大于当前 `期初余额 + 总进账金额` 能覆盖的金额，需要把期初余额修改为刚好满足最终余额的值。

建议修正公式：

```text
修正后的期初余额 = 支出合计 + 最后卡余额 - 收入合计
```

如果业务要求收入合计必须等于总进账金额，则使用：

```text
修正后的期初余额 = 支出合计 + 最后卡余额 - 总进账金额
```

修正后重新计算：

```text
总支出金额 = 修正后的期初余额 + 总进账金额 - 最后卡余额
```

### 最后卡余额强校验

最后卡余额不能少，也不能多。

生成并重新计算余额后，最后一条流水余额必须等于页面配置的最后卡余额。

```text
最后一条流水余额 == 最后卡余额
```

如果不相等，不能直接保存为成功结果。必须执行以下任一处理：

1. 调整期初余额。
2. 调整最后一条可移动支出或收入流水。
3. 重新生成流水。

推荐优先级：

1. 不超过总进账金额的前提下调整收入。
2. 调整支出。
3. 调整期初余额。
4. 重新生成。

## 余额滚动规则

流水生成后按 `AccountTime` 升序排序。

```text
当前余额 = 期初余额

for each 流水 in 按时间排序的流水:
    当前余额 = 当前余额 + 流水金额
    流水.Balance = 当前余额
```

最后一条流水的 `Balance` 必须等于最后卡余额。

## 字段映射规则

### 普通参照明细

来源为参照明细时：

1. 基础字段来自中间流水项。
2. 银行专属字段从当前参照明细行复制。
3. 使用当前银行的 `ReferenceColumns` 作为字段范围。

### 固定日期增加项

来源为固定日期增加项时：

1. 基础字段来自中间流水项。
2. 银行专属字段从当前固定项行复制。
3. 使用当前银行的 `ReferenceColumns` 作为字段范围。

### 利息流水

来源为利息时：

1. 基础字段来自利息占位项。
2. 银行专属字段从 `BankRateConfig` 复制。
3. 只复制当前银行流水字段中可配置的非金额、非日期字段。

### 系统字段自动补位

参照明细、固定日期增加项和利息配置只负责提供业务字段。生成流水时，如果当前银行流水字段包含系统字段，且字段为空，需要按银行规则自动补位。

1. `ID` 只作为列表行号，不写入银行业务字段；银行字段里的 `序号` 应写入业务序号。
2. 微信需要单独生成 `交易单号` 和 `商户单号`，不能使用普通流水号兜底。商户消费类交易使用 `4200...` 号段；扫码、二维码、微信支付、转账、提现、红包、群红包类交易使用 `1000...` 号段；转账类 `商户单号` 为 `/`，其它场景按对应交易类型生成。
3. 支付宝的 `流水号`、`商家订单号` 按支付宝流水字段生成。
4. 农行需要自动生成 `日志号`，并按交易场景生成 `附言` 前缀；已有 `UA`、`NA`、`NG` 前缀时不覆盖。
5. 中信凭证类字段使用 `ECPS + yyyyMMddHHmmss + 6位数字`。
6. 其它银行按字段名补齐 `凭证号`、`回单编号`、`全局路由号`、`外部系统流水`、`柜员号`、`网点号`、`应用号`、`交易代码`、`地区号`、`账户序号`、`分户序号` 等。
7. 自动补位只填空字段，不能覆盖参照明细、固定日期增加项或利息配置里已经提供的值。

## 本地库保存建议

生成保存时建议使用事务：

1. 开始事务。
2. 删除当前用户旧流水。
3. 插入新流水。
4. 更新用户的起止日期、期初余额、生成配置。
5. 提交事务。

如果任一步失败：

1. 回滚事务。
2. 弹窗提示生成失败。
3. 不打开流水明细页。

## 实现优先级

建议按以下顺序开发：

1. 建立中间流水模型。
2. 实现参照明细生成。
3. 实现固定日期增加项生成。
4. 实现余额滚动。
5. 实现收入合计限制。
6. 实现最后卡余额强校验。
7. 实现期初余额自动修正。
8. 实现利息占位与利息计算。
9. 实现银行字段映射。
10. 实现本地库事务保存。
11. 接入流水明细页面展示。

## 注意事项

- 参照明细和固定日期增加项是银行级数据，不是用户级数据。
- 用户只保存生成配置和生成后的流水。
- 收入金额用正数，支出金额用负数；统计支出合计时取绝对值。
- 金额生成、余额计算、利息计算不要用 `double`，应使用 `decimal`。
- 日期比较时要明确是否包含当天结束时间，结束日期建议统一补到 `23:59:59`。
- 所有保存操作必须有成功或失败提示。
- 不要依赖原程序的混淆类名，按业务语义重新实现。
