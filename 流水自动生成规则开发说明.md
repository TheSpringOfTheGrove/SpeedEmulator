# 真诚流水自动生成规则开发说明

本文档记录 `SpeedEmulator` 当前版本的流水自动生成规则，依据当前代码实现整理，主要对应：

- `Services/FlowAutoGenerator.cs`
- `Services/ZhenchengFlowGenerationAdapter.cs`（历史适配器，当前主生成流程不再调用）
- `ViewModels/FlowGenerationViewModel.cs`
- `Models/FlowGenerationModels.cs`
- `Models/FlowRecord.cs`
- `Models/BankUser.cs`
- `Repositories/InMemoryFlowGenerationRepository.cs`

约束：真诚程序只作为只读参考，不破解数据库，不复制反编译混淆代码。当前主流程已经改为 `FlowAutoGenerator.Generate` 本地复刻规则生成；`ZhenchengFlowGenerationAdapter` 仅保留为历史适配器代码，不参与页面“开始生成”。

## 当前主流程

1. 用户在用户列表选择用户后进入“自动生成流水”页面。
2. 页面从当前用户读取 `StartDate`、`EndDate`、`OpeningBalance`，写入生成配置的开始日期、结束日期、期初余额。
3. 页面从银行级流水生成配置读取参照明细、固定日期增加项目和金额配置。
4. 点击“计算”时，根据起止日期、总进账金额和最后卡余额计算总支出及月收入/月支出范围。
5. 点击“保存配置”时，保存当前银行的生成快照，并把开始日期、结束日期、期初余额同步回当前用户。
6. 点击“开始生成”时，先保存配置，再检查当前用户是否已有流水；已有流水时只询问是否覆盖。
7. 如果勾选自动计算利息，生成前必须能读取到有效利息配置或默认配置。
8. 当前版本调用 `FlowAutoGenerator.Generate` 本地复刻规则生成流水，不再调用真诚 DLL 或 `ZhenchengFlowGenerationAdapter.TryGenerate`。
9. 生成成功后，执行余额、利息和字段补全检查；检查不通过时只写状态提示，仍保存流水并打开流水明细页面。
10. 生成后的流水按用户保存，后续在流水明细页面查看、编辑、导入、导出。

## 数据归属

当前实现的数据归属如下：

- 用户级数据：用户信息、开始日期、结束日期、期初余额、是否自动计算利息、生成出来的流水记录。
- 银行级数据：参照明细、固定日期增加项目、流水生成配置快照、银行利息配置、字段设置。
- 流水记录：按 `BankId + BankUserId` 保存，是用户级结果数据。

注意：`IFlowGenerationRepository.SaveAsync(bankId, bankUserId, snapshot)` 虽然带 `bankUserId` 参数，但当前 `InMemoryFlowGenerationRepository.CreateKey` 实际只使用 `bankId`，因此参照明细、固定日期增加项目和金额配置快照都是银行级共享数据。

## 页面配置规则

### 开始日期、结束日期、期初余额

- 进入页面时从当前用户读取：
  - `BankUser.StartDate` -> `Config.StartTime`
  - `BankUser.EndDate` -> `Config.EndTime`
  - `BankUser.OpeningBalance` -> `Config.OpeningBalance`
- 保存配置或生成成功后，同步回当前用户。
- 结束日期不能早于开始日期；否则生成前直接提示并停止。

### 计算按钮

计算按钮当前公式：

```text
月份数 = 起始日期到结束日期覆盖月份数，包含首尾月
总支出 = max(0, 总进账金额 - 最后卡余额)
月均收入 = 总进账金额 / 月份数
月均支出 = 总支出 / 月份数

月收入最小金额 = round(月均收入 * 0.45, 2)
月收入最大金额 = round(月均收入 * 1.80, 2)
月支出最小金额 = round(月均支出 * 0.45, 2)
月支出最大金额 = round(月均支出 * 1.80, 2)
```

计算结果同时写入“按总进账金额方式分配金额”和“按月进账金额方式分配金额”两组月范围。

注意：当前计算按钮和本地生成器目标支出都不把期初余额计入总支出公式。期初余额只参与余额滚动和最终余额目标。

## 生成前校验

开始生成前必须满足：

- 已选择用户。
- 至少勾选一条参照明细或固定日期增加项目。
- 结束日期不早于开始日期。
- 如果当前用户已有流水，用户选择“是”才覆盖；选择“否”则取消本次生成。
- 如果当前用户勾选自动计算利息，必须有有效银行利息配置或默认配置。

生成结果出现负余额时，页面会提示是否执行负余额修正；选择“是”时必须修正所有负余额并继续保存流水，修正只调整流水时间、可选支出和可调金额，不修改期初余额。只有用户选择“否”时，才按原生成结果保存负余额并打开流水页面。期末余额偏差或利息校验不合格不会弹“余额修正失败”阻断窗口，只写状态提示并继续保存。

## 真诚完整黑盒调用规则（历史说明，当前停用）

旧适配方案曾使用 `ZhenchengFlowGenerationAdapter`，当前主流程已经停用：

1. 只取 `IsCheck=true` 的参照明细和固定日期增加项目。
2. 对参照明细和固定日期增加项目重新写入本次列表索引，便于真诚返回的计划项映射回本地规则。
3. 定位真诚运行时目录：
   - 环境变量 `SPEEDEMULATOR_ZHENCHENG_DIR`
   - 程序输出目录下的 `zhencheng-runtime`
   - 项目或当前目录下的 `VendorRuntime\Zhencheng`
   - 默认目录 `D:\真诚财务软件`
4. 调用真诚生成计划方法，拿到计划项后不直接保存，而是先经过本地规则修正和字段补全。
5. 如果真诚没有返回计划项，或运行时不可用，生成失败，不再使用本地随机兜底。

## 历史适配器预算规则（当前停用）

旧适配器曾把当前配置改造成更适合真诚生成的月明细预算：

```text
目标收入 = max(0, 总进账金额)
目标支出 = max(0, 总进账金额 - 最后卡余额)
目标期末余额 = 期初余额 + 最后卡余额
```

然后把配置强制转成按月明细模式传给真诚：

```text
Config.AllOutMoney = 目标支出
Config.SelectIndex = 2
Config.MonthGenData = 每月收入/支出预算
```

月预算生成规则：

- 如果页面已有月明细配置，先按每个月覆盖到的配置段金额求和，再按总收入/总支出等比例缩放。
- 如果没有月明细配置，按每个月在起止日期内覆盖的天数比例分配总收入/总支出。
- 月收入/月支出最小最大范围来自当前选择的金额模式。
- 如果月最小/月最大范围按月份覆盖比例计算后仍能容纳总额，则把每月目标限制在范围内，并把差额重新分摊。
- 如果范围数学上容不下总额，则保留原月目标，不强行套范围。

## 固定日期增加项目收入超出规则

固定日期增加项目优先生成，是必须保留的数据。若固定日期增加项目里的收入合计已经达到或超过页面总进账金额，参照明细里的收入可以忽略不生成。

```text
固定收入合计 >= 总进账金额:
    删除参照明细收入计划项
    收入目标 = 总进账金额
    支出目标 <= 实际收入合计
```

固定收入和参照明细收入都不能把收入合计抬高到页面总进账金额以上。若固定项目最小金额已经超过总进账金额，生成器会尽量把固定项目压到规则允许的最小金额，并继续优先保证余额不负和期末收口。

多出的收入优先通过支出流水消化，但支出合计也不能超过收入合计，最终尽量保证：

```text
最后一条余额 = 期初余额 + 最后卡余额
```

容差最多 1000。

## 参照明细生成规则

参照明细用于普通流水，只处理 `IsCheck=true` 的行。

### 每月出现次数

- `PercentMonth > 0`：每个月必须出现对应次数。
- 必须出现的参照明细不能因为负余额、配平或减少支出而被删除。
- 必须出现的记录金额仍必须在该行 `MinMoney` 到 `MaxMoney` 范围内。
- 必须出现的记录会尽量分散在该规则允许日期范围内，不集中在同一天。
- `PercentMonth = 0`：可选流水，可用于补足收入/支出，也可以在资金不足时减少或删除。

### 收入参照明细

当前规则会特别处理“收入参照明细很少”的情况：

- 当收入参照明细数量较少时，会把稀疏收入拆成更多笔，但每笔必须落在来源规则金额范围内。
- 拆分上限为每笔最多拆成 8 笔。
- 拆分后的日期按月份目标分配，优先选择非相邻日期，避免同一天或连续几天堆积。
- 如果有非支付类收入参照明细，优先用非支付类收入替换支付类收入。
- 支付类收入识别关键词包括：`代付`、`支付`、`付款`、`消费`、`红包`、`零钱提现`、`财付通`、`支付宝`、`微信`。

### 支出参照明细

支出参照明细生成后会参与余额稳定处理：

- 必须出现的支出不能删除，只能在自身金额范围内调整。
- 可选支出可以减少金额。
- 如果减少后仍无法避免负余额，可删除可选支出。
- 收入不够覆盖支出时，优先减少非必须出现的支出流水。
- 后续如果余额安全且支出目标差额较大，会尝试恢复或增加可选支出，但仍不能导致负余额。

## 固定日期增加项目生成规则

固定日期增加项目只处理 `IsCheck=true` 的行。

当前主流程先把固定项目传给真诚生成，再由本地规则做金额和字段收口。固定项目的基本约束：

- 金额必须符合该行 `MinMoney`、`MaxMoney`、`FloutLength`。
- 日期必须符合固定项目配置。
- `ReCnt` 作为重复次数，最小按 1 次处理。
- 固定项目是必须出现的数据，不能因为余额修正被删除。
- 固定收入超出总进账时，参照明细收入可以忽略，多出的收入通过支出流水消化。

本地保留的固定日期解析能力如下，供后续维护参考：

- 固定日按月生成。
- 固定项目会在目标日期内按 `StartDay`、`EndDay` 生成时间。
- 重复多次时会在基础时间上追加分钟，减少完全同时间。

## 金额范围和小数位规则

所有普通流水金额必须来自已勾选启用的参照明细或固定日期增加项目，并保持在对应来源行的最小金额和最大金额之间。

金额粒度由 `FloutLength` 决定：

- 空值：按 `0` 处理。
- `0`：取整数。
- `-1`：取整到十位。
- `-2`：取整到百位。
- `-3`：取整到千位，以此类推。
- `1`：保留 1 位小数。
- `2`：保留 2 位小数。
- 正数 `n`：保留 `n` 位小数。

四舍五入规则使用 `MidpointRounding.AwayFromZero`。在“向下缩减金额”场景会按粒度向下取整，避免缩减后超过可用余额。

最终显示和余额计算统一按两位金额处理。普通流水不允许保留 `0.00` 或 `-0.00`；本地生成器中的利息税 0.00 记录例外。

## 日期和时间分散规则

生成日期必须落在页面开始日期和结束日期之间。结束日期如果没有时间部分，按当天 `23:59:59.9999999` 处理。

时间范围规则：

- 使用来源规则的 `StartDay` 和 `EndDay` 作为小时范围。
- 小时范围被限制在 `0` 到 `23`。
- 如果结束小时小于开始小时，会交换。
- 时间会随机或按序列生成分钟、秒，尽量避免完全相同。

分散规则：

- 稀疏收入会按月分桶，优先放在未使用且不相邻的日期。
- 生成后会尽量补齐起止范围内没有流水的日期；补齐时只使用参照明细，不强造固定日期增加项目。
- 每日补齐记录优先选择能改善期末余额方向的收入或支出，并且金额、时间仍受来源参照明细约束。
- 同方向连续记录会反复尝试插入相反方向记录，或移动同方向记录位置。
- 同一月份同方向记录过密时，会计算每天允许的最大同方向数量，并把可移动参照明细挪到更稀疏的日期。
- 最终会去重到秒级，避免完全相同时间。
- 若规则日期范围太窄、固定项目过多或只有单一收入/支出来源，仍可能无法完全消除连续方向，但会尽量移动到可用范围内。

## 余额稳定和期末收口

余额滚动公式：

```text
当前余额 = 期初余额
按 AccountTime 升序、再按流水号排序:
    当前余额 = round(当前余额 + 交易金额, 2)
    当前流水余额 = 当前余额
```

目标期末余额：

```text
目标期末余额 = 期初余额 + 最后卡余额
```

当前容差：

```text
abs(最后一条余额 - 目标期末余额) <= 1000
```

注意：旧实现曾临时放宽到 10000，当前代码已按业务要求收回到 1000。

余额稳定处理顺序：

1. 对计划项按规则金额范围和小数位归一。
2. 补足每月必须出现的参照明细。
3. 如果固定收入已经覆盖总收入，删除参照明细收入计划项。
4. 拆分和分散稀疏收入。
5. 优先把支付类收入替换为非支付类收入。
6. 把收入合计归一到 `总进账金额`，不能超过页面配置的总收入。
7. 按实际收入合计计算目标支出：`max(0, 实际收入合计 - 最后卡余额)`。
8. 把支出合计归一到目标支出，同时保证 `支出合计 <= 收入合计`。
9. 对空白日期尽量补一条参照明细流水。
10. 如果滚动过程中出现负余额，优先把后续收入提前到问题支出之前。
11. 如果不能提前收入，尝试把问题前的可移动支出延后。
12. 如果仍不行，减少或删除问题前的可选支出。
13. 如果删除支出后支出合计差距较大，尝试在余额安全的位置恢复或增加可选支出。
14. 分散同方向连续记录，分散同一天过密记录，并去除重复秒级时间。
15. 创建真诚流水记录后，再执行最终余额收口。

最终余额收口最多尝试 18 轮：

- 如果存在负余额，先减少支出记录。
- 如果期末余额高于目标容差，增加支出或减少收入。
- 如果期末余额低于目标容差，增加收入。
- 收口过程中会先把收入压回 `总进账金额` 以内，再把支出压到不超过收入合计。
- 新增收口记录也必须来自已勾选规则，且遵守该规则金额范围和小数位。
- 新增收入补差不能超过 `总进账金额 - 当前收入合计`。
- 新增支出补差不能超过 `当前收入合计 - 当前支出合计`。
- 收口成功条件是期末余额落入 `目标期末余额 ± 1000`。

生成结果仍不满足余额或利息校验时，当前页面逻辑不会阻断保存，只会在状态栏写“自动放行”原因。

微信银行生成大量流水时，余额稳定、同方向分散、同日过密分散、秒级去重和最终补差循环会走较短上限，避免生成过程长时间卡死。最终补差的日期选择使用一次性滚动余额计算，不能在每个候选日期上重复扫描全量流水。该优化只限制循环次数和重复计算，不改变金额范围、日期范围和字段补全规则。

页面点击“开始生成”后，真正的流水生成计算必须放到后台线程执行；UI 线程只负责保存配置、更新进度、保存结果和打开流水明细，避免微信等大流水场景下窗口无响应。

## 利息生成规则

利息是否生成由当前用户的 `AutoCalculateInterest` 控制。

生成前规则：

- 如果用户未勾选自动计算利息，不生成利息。
- 如果用户勾选自动计算利息，必须能读取到有效银行利息配置或默认配置，否则生成前提示并停止。

主流程利息处理：

- 历史适配器曾优先调用真诚运行时的 `ProcessIntestData` 处理结息数据；当前本地生成器使用本地利息配置和日积数算法生成结息数据。
- 真诚返回后，本地识别结息流水并补齐字段。
- 结息识别文本包括利息配置中的结息摘要，或字段中包含 `结息`。
- 包含 `利息税` 的记录不按普通结息收入处理。

利息字段补全规则：

- 先应用银行利息配置中的字段值。
- 再从普通流水中统计常见字段作为默认值，例如应用号、序号、币种、钞汇、存期、约转期、地区号、网点号、柜员号、交易渠道、交易代码等。
- 工行利息默认值包含：应用号 `1`、序号 `0`、存期 `000`、约转期 `不转存`、通知种类 `0`、钞汇 `钞`。
- 非工行利息默认交易方式优先为 `转账`。
- 利息流水强制为收入，`IncomeFlag=C`，贷方金额为利息金额，借方为空。
- 利息行本方账号会按用户账号/卡号规则修正，避免卡号账号混淆。

本地生成器保留的利息算法：

```text
日积数 = sum(当日余额 * 持续天数)
利息 = round(日积数 * 年利率 / 36500, 2)
```

本地生成器会按利息配置的结息月份、结息日、起止小时生成结息占位；必要银行会追加 0.00 利息税记录。

## 用户账号和卡号规则

流水中的本方账号、卡号、账号列必须来自当前用户信息，不能从参照明细或固定项目乱带。

当前规则：

- 生成真诚用户对象时：
  - `Account`、`AccountNum` 优先取用户 `AccountNo`，其次取 `CardNo`。
  - `CardNum` 优先取用户 `CardNo`，其次取 `AccountNo`。
- 映射回本地流水时，如果 `FlowRecord.Account` 为空，按当前银行流水列名判断：
  - 列名更像卡号时优先 `CardNo`。
  - 列名更像账号时优先 `AccountNo`。
- 如果字段已有值且是当前用户账号/卡号中的另一个值，利息补全时允许替换成更符合当前银行列名的值。
- 银行用户字段中如果存在自定义列名，也会按列名识别 `卡号`、`账号`、`帐号`、`客户账号`、`账户号` 等取值。

## 字段补全规则

生成的流水先由真诚运行时映射出字段，再执行本地补全。

通用规则：

- 自动补位只填空字段，除农行附言特殊规则外，不覆盖参照明细、固定日期增加项目、真诚运行时已生成的有效值。
- `BankId`、`BankUserId` 缺失时用当前银行和用户补齐。
- `Currency` 缺失时使用用户币种，仍为空时用 `RMB`。
- `TradeCurrency` 缺失时使用 `Currency`。
- `IncomeAttribute` 按金额正负重写为 `收入` 或 `支出`。
- `CreditAmount`、`DebitAmount`、`IncomeFlag` 按金额正负补齐。

普通系统字段：

- 根据当前银行流水字段配置补齐流水号、凭证号、回单编号、全局路由号、柜员号、网点号、应用号、交易代码、地区号、账户序号、分户序号等。
- 中信凭证类字段使用 `ECPS + yyyyMMddHHmmss + 6位数字`。
- 支付宝、微信等特殊银行字段按各自字段规则生成。

## 外部系统流水号规则

当当前银行流水字段包含列名 `外部系统流水`，且记录被识别为消费类支出时，生成外部系统流水号。

格式：

```text
交易日期yyyyMMddHHmmss + 16位随机数字
```

例如：

```text
20250602194608 + 1234567890123456
```

消费类支出识别：

- 金额必须为支出。
- 文本字段包括摘要、业务类型、用途、交易说明、附言、交易地点、交易渠道、对方户名、商户名、界面等。
- 命中消费类关键词才生成外部系统流水号。

消费类关键词包括：

```text
消费、支付、快捷、无卡、财付通、支付宝、微信、云闪付、银联、刷卡、扫码、商户、京东、美团、拼多多、多多、抖音、淘宝、天猫、饿了么、生活缴费、手机充值、POS
```

非消费类关键词会排除：

```text
结息、利息、利息税、工资、他行汇入、汇入、转入、收款、转账、转支、转存、提现、取现、存款、柜面、冲正、冲销、贷款、还款
```

## 农行日志号和附言规则

农行包括银行名称包含 `农行` 或 `农业` 的银行。

### 日志号

如果当前农行流水字段包含 `日志号`，日志号按以下规则补齐：

- 2024-03-15 之前：9 位随机数字。
- 2024-03-15 含之后，2024-08-15 之前：`U + 9位随机数字`。
- 2024-08-15 含之后：`A-Z随机字母 + 9位随机数字`。

当前主流程使用本地规则生成日志号；真诚运行时只作为历史参考，不参与页面生成。

### 附言

如果当前农行流水字段包含 `附言`、`备注`、`留言`、`转账附言` 或 `APP备注`，会补齐农行附言流水号。

已有以下前缀时不重复覆盖：

- `UA + 16位数字`
- `NA + 31位数字`
- `NG + 31位数字`

附言前缀规则：

- 抖音类：`NA + yyyyMMdd + 23位随机数字 + 原附言`
- 代付且附言包含零钱提现：`NG + yyyyMMdd + 23位随机数字 + 原附言`
- 其他支付类：`UA + MMdd + 12位随机数字 + 原附言`
- 不属于上述场景：保留原附言文本。

支付类 `UA` 识别字段包括摘要、附言、交易渠道、业务名、业务类型、交易说明、对方户名、对方开户行、用途、界面等。

支付类 `UA` 关键词包括：

```text
财付通、微信、微信支付、扫二维码、二维码付款、支付宝、多多支付、拼多多、手机充值、特约商户
```

## 收入/支出方向识别

历史适配器中，收入参照明细主要按 `IncomeAttribute` 是否包含 `收入` 判断；当前本地生成器保留更宽松的收入/支出关键字识别。

本地生成器保留更宽松的方向识别：

- 明确支出关键词：`支出`、`出账`、`借方`、`扣款`、`付款`、`消费`、`转出`、`汇出`。
- 明确收入关键词：`收入`、`进账`、`入账`、`贷方`、`收款`、`转入`、`汇入`、`存入`。
- 如果贷方金额有值且借方为空，按收入。
- 如果借方金额有值且贷方为空，按支出。

维护参照明细时，应优先把 `IncomeAttribute` 配准确，避免生成方向错误。

## 保存规则

生成成功后：

1. 保存生成流水到当前用户。
2. 同步保存当前用户开始日期、结束日期、期初余额。
3. 弹窗提示生成成功和流水条数。
4. 打开流水明细页面。

生成失败只在以下情况发生：

- 没有用户。
- 没有勾选任何参照明细或固定日期增加项目。
- 结束日期早于开始日期。
- 覆盖已有流水时用户选择“否”。
- 自动计算利息但没有有效利息配置。
- 真诚运行时目录不可用。
- 保存流水或保存用户信息异常。

负余额会弹出负余额修正确认；系统优先修流水本身，不修期初余额。用户选择“是”时修正全部负余额并保存，选择“否”时才允许保存负余额。期末余额偏差或利息校验不合格不会阻断保存，只写状态提示并继续保存。

## 后续维护注意事项

- 调整金额规则时，必须同时考虑 `MinMoney`、`MaxMoney`、`FloutLength`、必须出现次数和固定项目不可删除。
- 调整余额规则时，注意当前目标是 `期初余额 + 最后卡余额`，容差是 `1000`。
- 调整收入规则时，固定日期增加项目收入优先保留；固定收入超出总进账时，参照明细收入可以不生成，但收入合计仍不能超过总进账金额，支出合计不能超过收入合计。
- 调整连续流水规则时，优先移动可选参照明细，不要破坏固定日期增加项目和每月必须出现次数。
- 调整农行字段时，日志号和附言前缀要优先保持真诚运行时结果；本地规则只做缺失或不完整时的补齐。
- 调整卡号/账号字段时，必须以当前用户信息为准，不能从参照明细的对方账号或规则默认值误填本方账号。
- 如果以后恢复真诚黑盒调用，需要同步更新本文档中“当前主流程使用本地复刻生成器”的说明。

## 2026-07-07 本地复刻生成规则（当前以本节为准）

当前主生成入口已经改为 `FlowAutoGenerator.Generate`，页面不再调用 `ZhenchengFlowGenerationAdapter.TryGenerate`。真诚程序和 DLL 只作为只读规则参考，不在用户点击“开始生成”时参与生成。

### 反编译参考结论

- 真诚生成计划入口按 `GenerateReference`、`GenerateConst` 和 `ComputeEntity` 生成候选计划。
- 固定日期增加项目先处理，按 `FixDay`、`FixType`、`ReCnt`、`StartDay`、`EndDay` 生成日期和时间。
- 参照明细只有 `PercentMonth > 0` 才是每月必出；`PercentMonth = 0` 或为空是可选流水。
- 每条金额都从来源行 `MinMoney` 到 `MaxMoney` 随机，并按 `FloutLength` 取整或保留小数。
- 收入累计超过 `AllInMoney` 时，真诚会停止继续追加后续收入计划。

### 当前本地生成顺序

1. 读取页面开始日期、结束日期、期初余额、总进账金额和最后卡余额。
2. 计算目标期末余额：`期初余额 + 最后卡余额`。
3. 计算目标支出：`min(总进账金额, max(0, 总进账金额 - 最后卡余额))`，确保支出不超过收入。
4. 按配置的三种模式生成月收入/月支出预算。
5. 优先生成已勾选的固定日期增加项目。
6. 再生成 `PercentMonth > 0` 的每月必出参照明细。
7. 用现有收入行的可调空间和 `PercentMonth = 0` 的可选参照明细补足总进账金额；如果可选收入来源不足，再用参照明细来源做可删除的补足行。
8. 按实际收入合计计算支出目标：`min(实际收入合计, max(0, 实际收入合计 - 最后卡余额))`，再用现有支出行和可选支出参照明细补足。
9. 如勾选自动计算利息，按本地利息配置生成结息/利息税，并补齐银行字段。
10. 最终收口优先调整已有可调流水金额和移动可移动流水时间，必要时删除可选流水，不在末尾集中追加一串同方向补差流水。

### 硬约束

- 普通流水只能来自已勾选启用的参照明细或固定日期增加项目。
- 金额永远不能超过来源行 `MinMoney` 和 `MaxMoney`。
- 小数位规则：空值按 `0`；`0` 取整数；`-1` 取十位；`-2` 取百位；`-3` 取千位；正数保留对应小数位。
- 固定日期增加项目优先生成；如收入容量不足，后续参照收入会被压缩或跳过。
- 每月必出参照明细优先保留；可选流水可减少、删除或不生成。
- 收入合计不能超过总进账金额。
- 支出合计不能超过收入合计。
- 期末余额目标为 `期初余额 + 最后卡余额`，校验容差为 `1000`。
- 日期必须落在页面起止日期内，时间必须落在来源行 `StartDay` 到 `EndDay` 小时范围内。
- 不允许生成普通 `0.00` 或 `-0.00` 流水；利息税 `0.00` 例外。

### 分散规则

- 生成时维护每天收入条数、支出条数、总条数和已占用秒。
- 选择日期时优先选择同方向少、邻近日同方向少、总条数少的日期。
- 可选流水按月预算分散生成，不把补差集中放在结束日或同一天。
- 总进账补足阶段仍按月预算比例分配，先补月份缺口，再把少量余差分散到完整起止区间。
- 负余额修正优先把问题发生前的可移动支出移到后面资金更充足、同方向更少的日期。
- 同一秒重复时会在来源规则允许的小时范围内重新取秒级时间。
- 如果配置本身只有单方向来源，连续同方向无法完全消除，但仍会尽量分散到不同日期和不同时间。

### 最终调平规则

- 先压收入：超过总进账金额时，优先减少或删除可选收入；必出/固定收入只能在来源金额范围内调整。
- 再补收入：收入不足总进账金额时，先增加现有收入行到来源上限，不足再按月分散新增参照明细补足行。
- 再压支出：支出超过收入时，优先减少或删除可选支出。
- 再补支出：按 `实际收入 - 最后卡余额` 计算目标支出，支出不足时先增加现有支出行，不足再按月分散新增可选支出。
- 出现负余额时，优先移动负余额出现前的可移动支出到后面资金充足的位置；移动失败后才减少或删除可选支出。
- 期末余额偏高时，优先增加已有支出金额；不够时减少已有收入金额。
- 期末余额偏低时，优先减少已有支出金额；不够时增加已有收入金额，但仍不能突破总进账金额。

### 字段补全

- 流水本方账号/卡号永远以当前用户信息为准，根据银行流水列名判断优先取 `AccountNo` 还是 `CardNo`。
- 普通字段仍从来源行复制，再按银行流水列名做别名回填。
- 外部系统流水号只给消费类支出生成，格式为 `yyyyMMddHHmmss + 16位随机数字`。
- 农行日志号和附言继续按本地复刻规则补齐：日志号按日期阶段生成，附言按 `UA`、`NA`、`NG` 前缀规则识别和拼接。
- 利息流水先应用利息配置字段，再从普通流水统计常见字段作为默认值，最后套用账号/卡号和银行字段补全规则。

## 2026-07-08 生成金额与必出保护补充

- 金额随机按来源行 `MinMoney` 到 `MaxMoney` 分三段取值：15% 偏小、70% 偏中、15% 偏大。
- 金额生成会优先避开同方向已经出现过的相同金额；如果来源行金额范围和小数位/取整粒度本身只有一个可用值，则允许重复。
- `FloutLength` 继续按来源行生效：空值按 `0`，`0` 取整数，`-1` 取十位，`-2` 取百位，正数保留对应小数位。
- 固定日期增加项目和 `PercentMonth > 0` 的参照明细仍优先生成；负余额修正只能把必出/固定支出降到来源最小金额，不能把它们降成 0 后删除。
- 最大金额拆分逻辑本版暂时停用，保留代码入口注释，避免把单方向流水拆得过密。
- 主生成随机种子加入运行时随机量，同一组配置每次点击生成都会重新抽取金额和时间。

### 2026-07-08 金额桶细化

- 对 `MinMoney >= 1000` 且 `MaxMoney >= MinMoney * 4` 的宽金额范围规则，不再用月预算平均值压低单笔上限，先按来源行完整范围抽取金额，再由后续总额收口调整。
- 15/70/15 的桶区间按完整范围切分：低段约为前 22%，中段集中在 35% 到 60%，高段从 67% 开始。
- 例如 `5000~50000`：低段约 `5000~14900`，中段约 `20750~32000`，高段约 `35150~50000`。

### 2026-07-08 必出次数修正

- `PercentMonth > 0` 的参照明细只在必出生成阶段按月份生成对应次数。
- 总额补差、期末余额强制补差和可选流水补充阶段只允许使用 `PercentMonth <= 0` 的可选参照明细新增流水，不能再拿必出参照明细补差。
- 后续负余额或总额修正可以调整必出流水金额，但不能新增额外同来源必出流水，也不能把必出流水删为 0。

### 2026-07-08 金额重复修正

- 最终保存前会对收入、支出两个方向分别做交易金额去重。
- 重复金额优先按来源行的金额粒度在 `MinMoney` 到 `MaxMoney` 内微调，例如一位小数规则会按 `0.1` 递增或递减。
- 如果某个来源行按 `MinMoney`、`MaxMoney` 和小数位计算后只有一个可用金额，则无法在不突破来源规则的情况下完全去重。
